# MVP & Deliverables

## Hackathon goal

The hackathon should produce a **focused MVP**, not a full platform. The expectation is a small but working end-to-end solution with a clear source basis, an initial data model, and visible value for prospective analysis.

## Minimum scope for the MVP

| Component | Minimum requirement |
| --- | --- |
| Data sources | integrate at least **4** publicly available data sources |
| Shared schema | cover **source, dataset, observation, indicator, geography, evidence, version** |
| Data storage | first **versioned** storage logic for raw or working data |
| Thematic structure | e.g. **PMESII-PT**, **PESTLE**, or a similar framework |
| User interface | dashboard or web app with **at least 1 map**, **1 timeline**, and **6 to 10 indicators** |
| Signals | trend cards or weak signals with **visible source references** |
| Documentation | short technical documentation on architecture, data sources, and reproducibility |

## Useful deliverables for submission

- runnable prototype or demonstrable build
- documented data sources with URL, format, access path, and retrieval date
- schema or data model for central entities
- documented ingestion and processing steps
- at least one clearly readable visualization with analytical value
- short description of indicator logic and its limits

## What should be visible in the result

- how different sources are related to each other
- why the selected indicators matter for strategic resilience or early detection
- how weak signals or trend changes are justified
- how sources, versions, and evidence remain visible in the product

## A strong hackathon scope usually means

- 4 to 6 clean sources instead of many half-integrated ones
- 6 to 10 well-explained indicators
- 1 to 2 robust signal or trend types
- 1 clearly documented path from retrieval to visualization

## Not required

- no complete operating model
- no perfect coverage of every topic
- no highly complex model landscape without visible user value
- no comprehensive automation of every possible source

!!! tip "Rule of thumb"
    A strong MVP is small enough to work reliably during the hackathon and strong enough to be reusable afterwards.
